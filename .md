# Multi-Store POS SaaS — Improvement Plan

A thorough review of the codebase reveals a well-structured foundation with real-time state management (Zustand), a clean glassmorphism design system, and a solid Prisma schema. Here are targeted improvements organized by impact and effort.

---

## 🔴 High-Impact (Fix Soon)

### 1. Security — Hardcoded Password Hash in Schema

> [!CAUTION]
> `schema.prisma` line 38 has a **hardcoded bcrypt hash** as the default `passwordHash` for all new users. This means every new user gets the same default password, which is a serious security risk.

**Fix**: Remove the `@default(...)` from `passwordHash`, and enforce password setting on user creation in the API route (`/api/users`). Generate a secure random temp password or require one on creation.

---

### 2. No Authentication Middleware / Route Protection

> [!CAUTION]
> All 12 API routes under `/api/` appear to have no authentication guard. Any user (or external caller) can `POST /api/sales`, `DELETE /api/held-sales`, etc. without being logged in.

**Fix**: Create a shared `lib/auth.ts` helper that reads a session token (e.g. from a cookie or `Authorization` header) and validates it before allowing API access. Apply it to every route handler.

---

### 3. No Loading / Error States on Data Fetch

> [!WARNING]
> `fetchData()` in `posStore.ts` silently swallows errors with `console.error`. The UI has no loading skeleton or error boundary, so users see a blank screen if the DB is unavailable.

**Fix**: Add `isLoading` and `error` fields to the `UISlice`. Show a loading skeleton on the main layout and a toast/banner on error.

---

### 4. `any` Types in Sidebar Props

> [!WARNING]
> `SidebarContentProps` in `Sidebar.tsx` uses `currentUser: any` and `currentStore: any`. This defeats TypeScript's safety net — a typo in property access won't be caught at compile time.

**Fix**: Import and use the `User` and `Store` types from `posStore.ts`.

---

## 🟡 Medium-Impact (Next Sprint)

### 5. Currency is Hardcoded as "GHS"

The currency symbol `GHS` appears ~30 times across views hardcoded as a string literal. The `Store` model already has a `currency` field.

**Fix**: Create a `useCurrency()` hook that reads `currentStore.currency` from the store, then use it as a formatter throughout all views. This makes the app truly multi-currency.

---

### 6. Dashboard Trend % is Fake

In `Dashboard.tsx`, `StatCard` is rendered with hardcoded `trend={12.5}` and `trend={8.2}`. These numbers are not computed from real data.

**Fix**: Compute yesterday's revenue and compare to today's for a real percentage change. Same for transaction count.

---

### 7. Zustand `persist` Persists Sensitive Data

`posStore.ts` uses `persist({ name: "multipos-storage" })` with **no partition** — this saves `currentUser`, `sales`, `products`, `customers`, and more to `localStorage`. On a shared kiosk device this is a serious data-leak vector.

**Fix**: Use `partialize` to only persist `currentStoreId`, `sidebarOpen`, and `activePage`. Re-hydrate real data from the API on every login.

```ts
// posStore.ts
persist((...a) => ({ ... }), {
  name: "multipos-storage",
  partialize: (state) => ({
    currentStoreId: state.currentStoreId,
    sidebarOpen: state.sidebarOpen,
    activePage: state.activePage,
  }),
})
```

---

### 8. POS Page Has No Barcode Scanner Support

The search box in the POS filters by name/barcode typed manually, but doesn't auto-submit when a barcode scanner fires (which ends input with `\n` or `Enter`).

**Fix**: Add an `onKeyDown` handler on the search input that, when `Enter` is pressed and the search string matches exactly one product, adds it to cart and clears the input.

---

### 9. No Pagination on Tables

`ProductsPage`, `SalesPage`, `SuppliersPage`, and `CustomersPage` render entire arrays into `<table>` — with enough records this will freeze the browser.

**Fix**: Add a simple `usePagination` hook (page, pageSize, total) and render only the current page slice. Add Prev/Next controls to table footers.

---

### 10. Missing `HeldSale` ↔ `Store` Relationship

In `schema.prisma`, `HeldSale` has no `storeId` field. When a super admin switches stores, all held sales from all stores are mixed together.

**Fix**: Add `storeId String` + `store Store @relation(...)` to `HeldSale` and `HeldSaleItem`, and filter held sales by store in `salesSlice.ts`.

---

### 11. Receipt Modal — No Print Layout

The `ReceiptModal` component exists but `globals.css` has a very blunt `body > * { display: none; }` print rule. This likely doesn't target the modal DOM correctly.

**Fix**: Give the receipt container a stable class (e.g. `receipt-root`) and use CSS `@media print` to show only that element. Also add a `window.print()` call button.

---

### 12. Emoji Characters in Filter Buttons

`ProductsPage.tsx` line 118 still has raw emojis: `⚠ Low Stock`, `❌ Out of Stock`, `📅 Expiring`. These were mentioned as a previous task (conversation b28056fc) that may not be fully done.

**Fix**: Replace with Lucide icons (`AlertTriangle`, `XCircle`, `CalendarClock`).

---

## 🟢 Quick Wins (Low Effort, High Polish)

### 13. Add `<title>` and `<meta description>` per Page

`layout.tsx` has a static title. Each page (`Dashboard`, `POS`, `Reports`, etc.) should set its own document title dynamically for better UX and SEO.

```ts
// In each view component:
useEffect(() => {
  document.title = "Dashboard — MultiPOS";
}, []);
```

---

### 14. Empty State Improvements

Most pages show a plain text "No data" message. Add illustrated empty states with an icon, a headline, and a CTA button (e.g. "Add your first product →").

---

### 15. `cn()` utility — Verify it Exists

The codebase imports `cn` from `../../utils/cn` — make sure this file exists and properly merges Tailwind classes using `clsx` + `tailwind-merge` (both are already in `package.json`).

---

### 16. Add `react-hot-toast` or Similar for Action Feedback

Currently, all API actions (add product, complete sale, delete) give no user feedback on success or failure. A simple toast notification system would dramatically improve UX.

---

### 17. `SaleItem.productId` is Not a Real FK in Prisma

`SaleItem` has `productId String` but no `@relation` to `Product`. If a product is deleted, `SaleItem.productId` becomes a dangling reference.

**Fix**: Add `product Product @relation(fields: [productId], references: [id])` to `SaleItem`, and use `onDelete: Restrict` to prevent accidental product deletion with sales history.

---

### 18. Supplier Balance is Never Updated

`Supplier.balance` is initialized to `0.0` but there is no API logic that updates it when a `PurchaseOrder` is received or paid. The field is effectively dead data.

**Fix**: When a purchase order status changes to `"received"`, add the order total to `supplier.balance` in the API handler.

---

## 🔵 New Features to Consider

| Feature                                    | Rationale                                                                                                                                                        |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Dark Mode**                              | The glassmorphism design system could support a dark variant with CSS variables — just flip the background gradient and glass opacities.                         |
| **CSV/PDF Export**                         | Add export buttons to Reports and Sales History pages. `jsPDF` + `papaparse` are lightweight additions.                                                          |
| **Real-time Sync (Polling or WebSockets)** | Multiple cashiers at the same store would overwrite each other's data. Add a 30s polling interval on `fetchData()` or use a Neon/Supabase realtime subscription. |
| **Customer Loyalty Points**                | `Customer.creditBalance` exists — build UI to redeem it as a discount during checkout.                                                                           |
| **Shift Management**                       | Track cashier start/end time and sales per shift. Essential for accountability in retail.                                                                        |
| **Stock Transfer Between Stores**          | The `InventoryLog.type = "TRANSFER"` is modeled but no UI exists for it.                                                                                         |
| **Low Stock Email Alerts**                 | Trigger an email (via Resend or SendGrid) when a product hits its `lowStockThreshold`.                                                                           |
